# 反思报告

> 本报告由本人撰写，使用 AI 助手做了语病和措辞润色。

## 1. 缘起：从食堂丢失的学生证说起

我选这个题，是因为我自己在校园里实在丢了太多东西。最离谱的一次是上学期初，我把学生证和身份证一起弄丢了，急急忙忙跑去补办；结果到了学期末，宿舍阿姨告诉我食堂的人员之前来宿舍楼问过，原来证件一直就在食堂。那一瞬间我意识到：南大跨着 4 个校区，每天都有这样的"东西其实就在隔壁，但失主从来不知道"的事情发生，而我们目前依赖的渠道——校内论坛、表白墙、各种群——信息会被无差别地刷掉，不要说跨校区，同校区的匹配几乎都是零。

所以这次期末项目我没有做"AI 原生应用"那一类炫技题，而是认真想做一个**校园失物招领平台**：四个校区共用一个池，按物品种类、时间、地点筛选，发帖匿名但又能跨设备查到自己发过的内容。30 秒一句话讲完它的存在意义：**让那个食堂阿姨问的"是谁丢的"问题，不再要等一个学期才传到我耳朵里**。

## 2. Superpowers 用下来：哪些是放大器，哪些是形式

整套 Superpowers 流程跑下来，让我最有"被兜底"感觉的环节是 **brainstorming 反问**。它不是一上来就替我写 spec，而是分块抛出设计草案让我逐节签字，并且会反过来追问一些我自己根本没意识到的问题。比如它提醒"列表返回是否要带 `contact_detail`？这是隐私字段"，我才把列表 schema 和详情 schema 拆开。这种"在我即将犯错之前把我拽住"的体验，是我以前自己写 spec 从来没有过的。

`writing-plans` 把每期 SPEC 拆成 8–11 个 2–5 分钟级 task，每个 task 都明确写出"先写哪条失败测试 + 跑哪条命令验证绿"。这让我对每一期的工作量有了真正的量化预期，而不是凭感觉估"大概一晚上吧"。`using-git-worktrees` 也很实用，我每期都开一个独立 worktree，主线在合并之前永远是干净的。

至于"形式大于实质"的环节，老实说我没有特别强烈的吐槽——每一项跑下来都有用。**唯一名实不符的，是 Superpowers 假设我能开真正的多 subagent**，这一点放到 §6 再展开。

## 3. TDD 在 AI 协作下：被推着走更稳

我做之前对强制 TDD 是有点抵触的，觉得"AI 都能直接生成对的代码了，再写一遍测试是不是冗余"。做完之后立场反转。

最有说服力的一次是：第一期我把"地点"当成自由文本字段实现，写了一批针对自由文本的测试。后来在 brainstorming 阶段把地点改成 4 校区枚举，对应的 schema 校验、表单组件全都改了。我以为改完就行，结果运行 pytest 时一票旧用例瞬间变红——那些旧测试仍在传"逸夫楼 B201"之类的字符串。那一刻我突然明白 TDD 在 AI 协作下的真正价值：AI 改代码很快，但它改的时候不会主动告诉你"喂，你前面还有 6 个用例假设了旧行为"。如果不是 TDD，我可能要等部署到线上、用户实际填表的时候才发现问题。

它也不是没有副作用。视觉重构那一期我承认自己偷偷绕开了"先写失败测试"，因为给"网格断点"或"卡片 hover 阴影"写测试的代价远高于直接看一眼页面。这件事我在 `AGENT_LOG.md` 里写了偏离声明。我的体感是：**逻辑层 TDD 是放大器，视觉层强制 TDD 是负担**。

## 4. subagent 颗粒度与 SPEC/PLAN 的双重身份

PLAN 把每期拆 8–11 个 task 这个颗粒度，对我来说**正好**。再大就容易跑歪——我尝试过把一个"App.vue Hero 重排"的 task 留得很大，结果 AI 一边改样式一边顺手"优化"了几个不相关的事件回调，回头我得逐行 diff 才能看明白它动了什么。再小就琐碎到没有信号价值——比如硬要把一个 5 行的工具函数拆成两个 task，反而打断了上下文。

我对 SPEC 与 PLAN 的看法：它们对我而言**既是给 AI 看的，也是给我自己看的**。每次在 AI 真正动笔之前，我都会回去把 SPEC + PLAN 通读一遍，这样如果发现里面有偏差，就可以**在生成代码之前**纠正，而不是等代码生成完再返工。返工要改的不只是代码，还有连带的测试、文档、commit 历史，成本要大得多。

举一个 SPEC 写得不够细导致小翻车的例子：用户系统的 SPEC AI只写了"用户名 3–32 位字符 / 密码 ≥6 位"，没有展开规定"校验失败时应该告诉用户具体哪一条不通过"。AI 实现出来的版本是直接 422 报错并显示笼统的"注册失败"，浏览器控制台里能看到原始 detail，但页面上只有一行没头没脑的红字。我把这次反馈贴回来后，给 AuthDialog 加了前端预校验和 422 数组型 detail 的解析。这件事教我的教训是：**SPEC 里的"错误处理"一节不能只写 HTTP code，必须连带写"用户在界面上看到什么"**。

## 5. 我的 prompt 模板：先轮廓，再细节

跟 AI 协作几十轮后，我形成了一个非常朴素但好用的 prompt 模板：**先讲需求大概，再详细描述四件事——页面上有哪些内容、点击按钮后发生什么、跳转去哪里、失败的话怎么办**。这四个问题听起来像产品经理写需求文档的标准段落，但放到 AI 协作里特别管用，因为它们恰好对应 AI 最容易"想当然"的四个边界：UI 元素、事件、路由、错误态。把这四件事讲清楚，AI 几乎不会跑偏。

另一个有效模板是冷启动验证那次写给 Cursor 的 prompt：要求每轮回复必须按"我理解的目标 / 我准备做的下一步 / 需要确认的疑问"三段式输出。这种结构会强制 AI 在"凭猜测继续"之前主动暴露不确定性。我打算把这条习惯带到日常开发里，无论是不是冷启动。

## 6. Open Design 与"AI 一眼能看出"的视觉

视觉重构这一期，我最终没有真的运行 Open Design 的 CLI/MCP——本地环境没把 daemon 跑起来，open design一直拉不下来。所以选择了"离线借用方法论"路线：写一份 `DESIGN.md` 当作 brand-spec，按它列的反 AI-slop 5 维 + P0/P1/P2 自评清单逐项重构组件视觉。

这条路线带给我的真正收获是**强迫我先把视觉决定写下来再去实现**——色板、字号节奏、间距台阶、圆角与阴影都用 CSS 变量 token 化，落到 `tokens.css`，组件里不允许出现硬编码颜色。这一步把"视觉一致性"从靠记忆和品味变成了靠规则。最终页面跑出来我自己挺满意的，跟"人做的"差不多——当然也可能因为我平时做前端比较随便，对 AI 出品的容忍度较高。


## 7. 重做一次会改什么 + 对方法论的批判

如果重做一次，我最想改的是**先做用户系统再做发帖功能**。用户系统是几乎所有功能的依赖：发帖要绑作者、点赞要标识、评论要显示作者、编辑/删除要鉴权。我现在的执行顺序是反过来的——前 4 期一直用 `X-Anon-Id` 做匿名身份，到了第 6 期才换成真用户。结果就是要清空数据库、改 60 个旧测试、把所有写操作的 fixture 重写一遍。这是典型的"先种树再迁树"——树长大了再迁，根系都得动。

对 Superpowers 方法论，我最大的批判是：**它默认你的环境支持开多个真正的子代理**。在它的设计哲学里，subagent-driven 是第二阶段评审的核心，"主代理 + 多个新鲜子代理"才能形成"自己审自己"以外的客观反馈。但我用的工具基本不允许真正的子代理派发，最终我只能在主会话里连续切换角色 + 任务间自审，这等于把"两阶段评审"降级成了"自我重读"。我能发现的问题主要是机械性的——命名、缺失断言、漏分号——架构层面的偏差很难靠自我重读发现。冷启动验证（让 Cursor 接手 task 2）是这次唯一真正接近"客观反馈"的环节。

它也假设 TDD 适用于所有变更类型——视觉层的偏离声明就是用来弥补这个假设的。还假设你愿意为每次偏离写理由：这条本身没问题，但如果偷懒不写，AGENT_LOG 就会跟现实脱节，方法论也就只剩了文件名。

## 8. AI 时代的工程师：从写代码的人到提需求的人

做完这个项目，我对"AI 时代的工程师价值在哪里"形成了一个比较朴素的判断：**未来的程序员会越来越像 PM**。具体来说，了解一些技术栈仍然是必要的——你得知道前端、后端、数据库大致是怎么协作的，否则连"哪里出了问题"都看不出来——但**最关键的能力是知道怎么提准确的需求**。

我的体感是：AI 已经把"写代码"这件事的成本压到了惊人地低，自然语言写代码很可能在不远的将来变得像 C 语言取代汇编一样普遍——AI 就是新一代的"编译器"。当代码廉价，瓶颈就移到上游：你知道**做什么**吗？你知道**做对了吗**？这两个问题，brainstorming 帮不了你最后一公里、TDD 帮不了你判断"做的事本身对不对"、Open Design 帮不了你定义产品。这三件事仍然是属于"人"的判断，正是这门课最后一段要我们认识的事情。
